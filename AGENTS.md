# 🤖 Правила и гайдлайны для AI-агентов (AGENTS.md)

> **ВНИМАНИЕ:** Этот файл — **основной источник истины** по правилам кода, стилю
> и архитектурным ограничениям для всех разработчиков и AI-агентов проекта
> `ttg-club-frontend`.
>
> Проект — это **Vite 5 + Vue 3** SPA (НЕ Nuxt). Правила ниже адаптированы именно
> под этот стек. Не переноси сюда допущения из Nuxt-проектов (auto-imports Nuxt,
> `@nuxt/ui`, Tailwind, `<ClientOnly>`, domain-алиасы вида `~feature` и т.п.).

## 🔮 Умный поиск контекста (протокол против дублирования)

Перед созданием ЛЮБОГО нового composable, утилиты или хелпера ты **ОБЯЗАН**:

1. **Проверить auto-imports.** Многое импортируется автоматически (без явного
   `import`). Источник истины — сгенерированный файл
   `src/shared/types/generated/auto-imports.d.ts`. Авто-импортируются: `vue`,
   `vue-router`, `pinia`, `@vueuse/core`, `@vueuse/math` и часть composables
   `naive-ui` (`useDialog`, `useMessage`, `useNotification`, `useLoadingBar`).
   Компоненты `naive-ui` — авто-импорт через резолвер (см.
   `src/shared/types/generated/components.d.ts`).
2. **Искать существующее.** Прежде чем писать своё, ищи по:
   - `src/shared/composable/` — общие composables (`useFilter`, `usePagination`,
     `useSearch`, `useLocalforageItem` и т.д.);
   - `src/shared/utils/` — общие утилиты;
   - `@vueuse/core` / `@vueuse/math` и `lodash-es` (`dayjs` — для дат).
3. **Логика действия:**
   - _Нашёл:_ используй существующее. Не создавай обёртку!
   - _Не нашёл:_ только тогда бери из `lodash-es`/`@vueuse` или создавай новый
     файл в правильном слое (см. «Стратегия переиспользования»).

## 🏗️ Архитектура и слои

Проект построен на слоях (enforced через `eslint-plugin-boundaries`). Правило
импортов: **слой может импортировать только из себя и слоёв левее**.

```
shared → features → layouts → pages
```

`src/core/` — точка сборки приложения (bootstrap): `main.ts`, `App.vue`,
`router.ts`, `store`, `config`. Обычные фичи в него не тащим.

### Структура

- **`src/shared/`** — переиспользуемое ядро без доменной привязки:
  `api`, `composable`, `config`, `const`, `directives`, `stores`, `types`,
  `ui`, `utils`.
- **`src/features/<feature>/`** — доменные фичи. Внутренняя структура по
  необходимости: `api/`, `model/` (constants, types, логика), `store/`,
  `types/`, `ui/` (компоненты фичи). Простые фичи могут быть плоскими (только
  `.vue`-файлы) — не усложняй без нужды.
- **`src/pages/`** — страницы (роуты). Композиционную логику пиши прямо на
  странице, не создавай пустой компонент-обёртку ради обёртки.
- **`src/layouts/`** — лейауты.

### Barrel-файлы (`index.ts`)

- Barrel-файлы **разрешены и используются** — как в корне фичи
  (`src/features/subscription/index.ts`), так и в подпапках (`model`, `api`,
  `ui`). Это норма проекта.
- Приоритет — импорт через barrel: `@/features/subscription/model`, а не
  `@/features/subscription/model/constants`.

### Импорты

- **Между слоями/фичами:** используй алиас `@/`:
  `import { ... } from '@/features/subscription/model';`,
  `import { SvgIcon } from '@/shared/ui/icons/svg-icon';`.
- **Внутри одной фичи** (между `ui` и `model` и т.п.) — относительные пути:
  `import { ... } from '../model';`.
- **В той же папке** — относительные: `import { ... } from './types';`.
- **`import/no-cycle`** включён (maxDepth 1) — циклических зависимостей быть не
  должно.

### Стратегия переиспользования

1. **UI:** 1) `naive-ui`; 2) `src/shared/ui`; 3) `src/features/*/ui`.
2. **Логика:** глобальные утилиты/сторы — в `src/shared/`; доменная
   логика/типы — в `src/features/*/model`.
3. **Типы:** доменные — в домене (`features/*/types` или `model`); глобальные —
   в `src/shared/types`.

## 🛠️ Стек и лучшие практики

- **Фреймворк:** Vite 5 + Vue 3 (Composition API, всегда
  `<script setup lang="ts">`).
- **Состояние:** Pinia. Избегай `provide/inject` для состояния — используй
  Pinia или composables.
- **UI-кит:** `naive-ui` (авто-импорт компонентов и части composables).
- **HTTP:** `axios` (общий клиент — в `src/shared/api`).
- **TypeScript (пиши строго, хотя `strict: false` в конфиге):**
  - `any` — избегай (линтер не запрещает, но это нежелательно). Типизируй явно.
  - Type assertions (`as Type`, `as unknown as Type`) — **нежелательны**
    (`consistent-type-assertions: never`). Предпочитай type guards и явную
    типизацию.
  - Для утилит со сложными типами задавай явный Return Type.
- **Внешние данные (API, ввод пользователя)** валидируй перед использованием, не
  доверяй форме ответа вслепую.
- **JSDoc — на русском.** Все нетривиальные функции (особенно в `utils`) снабжай
  JSDoc на русском языке. HTML-теги в JSDoc запрещены — используй Markdown.

## 🔄 Реактивность и поток данных

- **Иммутабельные трансформации:** при выводе нового состояния из объектов
  предпочитай `.map()` + spread (`{ ...item, selected: value }`) вместо
  `cloneDeep` + мутация. Прямой `cloneDeep` с последующей мутацией
  (`clone.field = x`) в утилитах — запрещён. `cloneDeep` допустим, только когда
  клон передаётся в функцию, возвращающую новый иммутабельный результат.
- **Гигиена watcher'ов:**
  - Если несколько watcher'ов вызывают **один и тот же** обработчик — объедини в
    `watch([sourceA, sourceB], handler)`.
  - Опасайся **циклов watcher'ов** (`watch(A) → меняется B → watch(B) →
    меняется A`). Всегда документируй guard, разрывающий цикл (например, проверку
    `isEqual`), комментарием — _почему_ цикл завершается.
- **Защита от `route.query`:** значения из `route.query` имеют тип
  `string | null | (string | null)[]`. Всегда проверяй и `null`, и пустую строку
  (`''`). Одной проверки `typeof val === 'string'` НЕ достаточно — добавляй явную
  проверку на истинность: `typeof val === 'string' && val`.

## 🧩 Гигиена composables

- **Чистые функции — вне замыкания:** если функция внутри composable НЕ
  обращается к реактивному состоянию, ref'ам или переменным из его области
  видимости, её **нужно** вынести:
  - в тот же файл, но **выше** функции-composable (module-level), ИЛИ
  - в `utils`-файл, если она переиспользуема.
- **Без мёртвых fallback'ов:** не пиши дефолты для значений, которые уже
  типизированы как обязательные (`group.type || 'filter'`, когда `type` не
  опционален). Это маскирует ошибки в данных API.
- **Строгое сопоставление, не «на глазок»:** при матчинге сущностей используй
  строгие критерии с чётким приоритетом:
  1. Точное совпадение по составному ключу.
  2. Однозначный fallback (единственная запись с нужным ключом).
  3. Верни `undefined` — не откатывайся на совпадение по одному полю (например,
     только по `name`), которое даёт false positive.

## 📝 Шаблоны Vue и обработчики событий

- **Порядок блоков в SFC:** `script` → `template` → `style` (enforced).
- **Гигиена шаблонов:**
  - **Запрещены** многострочные inline-обработчики. Если обработчик не влезает в
    одну строку или содержит логику — вынеси функцию в `<script setup>`.
  - _Плохо:_ `@update="v => { form = { ...form, val: v } }"`
  - _Хорошо:_ `@update="handleUpdate"`
- **Computed для сложной логики:** тернарники, вычисление динамических
  классов/цветов — переноси в `computed`. Шаблон должен быть декларативным.
- **`props`/`emits`** объявляй type-based (`defineProps<...>()`,
  `defineEmits<...>()`), используй `defineOptions`.
- **Инлайновые статические стили** запрещены (`vue/no-static-inline-styles`),
  биндинг `:style` — разрешён.

## ✍️ Нейминг

- **Язык:** только полные английские имена для переменных, функций, классов.
- **Ясность важнее краткости:** без аббревиатур (`lst`, `ua`) и абстрактных имён
  (`data`, `item`). Разные корни для разных смыслов (не путай `date` и `data`).
- **Имена компонентов Vue:** минимум два слова (`UserCard.vue`, `NavProfile.vue`;
  enforced `vue/match-component-file-name`). Шаблонные теги — `kebab-case`.
  Исключение — компоненты внутри `src/pages`.
- **БЕЗ ОДНОБУКВЕННЫХ ИМЁН:** `v`, `e`, `d`, `t` в аргументах, props, колбэках и
  верхнем скоупе **запрещены**. Исключение: `i` — только как индекс в обычном
  `for`.
- **Константы:**
  - **Без хардкода:** строки, ссылки, числа, конфиги не хардкодим в компонентах —
    выноси в `const`/`constants.ts` (`src/shared/const` или `features/*/model`).
  - **Ключи хранилищ (localforage/localStorage):** формат `domain:key-name`
    (kebab-case), например `tokenator:background-color`.

## 🎨 Стили (SCSS)

- **SCSS + `scoped`** в SFC. Переменные из
  `@/assets/styles/variables/index.scss` подключаются автоматически (не нужно
  импортировать вручную). Миксины подключай через
  `@use '@/assets/styles/variables/mixins' as *;`.
- **Именование — BEM:** `.block__element`, `&_modifier`, состояния `&.is-active`.
- **Цвета и токены — через CSS-переменные темы** (`var(--text-b-color)`,
  `var(--hover)` и т.п.). Не хардкодь цвета (`#fff`, `red`) — используй
  переменные темы.
- **Порядок свойств** контролирует `stylelint-config-clean-order` — не
  расставляй вручную, доверься автофиксу.

## 🖼️ Иконки

- Иконки — SVG-спрайт (`vite-plugin-svg-icons`) из `src/assets/icons/svg`.

  `<svg>` и не подключай сторонние icon-наборы.

## ✅ Команды и финальная проверка

**Команды (npm, Node ≥ 20.15):**

- `npm run serve` — dev-сервер (Vite, с host)
- `npm run build:dev` / `npm run build:prod` — сборка
- `npm run type-check` — проверка типов (`vue-tsc`)
- `npm run lint:write` — ESLint с автофиксом
- `npm run lint:check` — ESLint без фикса
- `npm run stylelint:write` / `npm run stylelint:check` — Stylelint
- Pre-commit: `lint-staged` (ESLint + Prettier для кода, Stylelint для стилей)

**Коммиты:**

- Формат [Conventional Commits](https://www.conventionalcommits.org/):
  `<type>[scope]: <описание>` (`feat`, `fix`, `refactor`, ...). Описание — на
  русском (как в истории репозитория).

**Финальная проверка (обязательно в конце задачи):**

1. `npm run lint:write` — стиль и автоформат.
2. `npm run stylelint:write` — стили.
3. `npm run type-check` — отсутствие ошибок типизации.
4. Дай краткое «Резюме изменений» (Где / Почему / Результат), когда тебя просят
   проверить/починить/отрефакторить.
